# Strength Program Calculator Style Guide

## 1. Общие принципы
- **Минимализм и Apple-like стиль.** Каждая страница и компонент должны напоминать гайдлайны Apple Human Interface Guidelines и использовать ощущение «воздуха» и простоты.
- **Модульность кода.** Все сущности (UI-компоненты, калькуляторы, вспомогательные утилиты) разделяются на небольшие модули с чёткой ответственностью.
- **Читаемость и простота.** Предпочитаем простые решения и понятные конструкции. Имя любого файла, функции или переменной должно раскрывать назначение без лишних комментариев.

## 2. Структура проекта
```
strength-program-calculator/
├── index.html          # Точка входа, подключает все модули
├── styles/             # Все CSS-файлы проекта
├── js/                 # Корневой каталог JavaScript-модулей
│   ├── calculators/    # Чистые функции расчётов
│   └── ui/             # Модули компонентов интерфейса
└── STYLEGUIDE.md       # Текущий документ
```

## 3. Правила именования
- **Файлы:** только `kebab-case`, напр. `week-card.js`, `program-form.css`.
- **Функции:** только `camelCase`, напр. `calculateVolume`, `renderWeekCard`.
- **Модули:** название описывает единственную ответственность (`progression-calculator.js`, `week-card.js`).
- **Calculators:** содержат **только чистые функции** — никакой работы с DOM или побочных эффектов.

## 4. Правила JavaScript
- Используем ES6-модули через `<script type="module">` в `index.html`.
- **Запрет на глобальные переменные.** Всё должно быть инкапсулировано внутри модулей и экспортироваться явным образом.
- Один модуль — одна задача: если модуль отвечает за карточку тренировки, он не должен выполнять расчёты.
- **UI-логика** (DOM-операции, управление состоянием интерфейса) живёт в `js/ui/` и импортирует расчётные функции из `js/calculators/`.
- Общие хелперы располагаются в корне `js/` или в подпапках, но не ломают принцип единственной ответственности.

## 5. Правила CSS
- Поддерживаем минималистичный, «чистый» Apple-like стиль: крупные заголовки, много свободного пространства, чёткая иерархия.
- Базовый системный стек шрифтов:
  ```css
  font-family: -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif;
  ```
- Используем базовую сетку (flex/grid), мягкие тени, аккуратные карточки, спокойную цветовую палитру, без визуального шума.
- Никаких внешних CSS-фреймворков (Bootstrap, Bulma, Tailwind и т.д.).
- Общие стили в `styles/base.css`, специфические стили компонент в отдельных файлах `styles/components/` (подкаталог создадим при необходимости).

## 6. Правила UI-компонентов
- **Карточки недель.** Содержат название недели, ключевые показатели (объём, интенсивность), списки тренировок. Выравнивание и отступы одинаковы во всех карточках.
- **Карточки тренировок.** Описывают упражнения, подходы, веса, комментарии. Используем однородные блоки и чёткую типографику.
- **Формы.** Минимум полей на экран, плавные состояния фокуса, понятные подписи и подсказки. Все поля выравниваются по сетке.
- **Таблицы.** Только когда данные действительно табличные. Используем светлые разделители, по возможности альтернативные фоны строк.
- **Адаптивность.** Интерфейс корректно работает от мобильных до десктопов: карточки выстраиваются в колонку на узких экранах, формы и таблицы не выходят за пределы экрана.

## 7. Правила расширяемости
- **Новые схемы тренировок.**
  1. Добавляем новый файл модуля в `js/ui/` для отображения схемы (например, `js/ui/powerlifting-template.js`).
  2. Если требуются новые расчёты, создаём чистый модуль в `js/calculators/`.
  3. В `index.html` подключаем новый UI-модуль и регистрируем его в соответствующем контроллере.
- **Новые калькуляторы.**
  1. Создаём файл в `js/calculators/` с чистыми функциями (`macro-cycle-calculator.js`).
  2. Пишем тестовые вызовы внутри модуля UI или в отдельном файле, но никогда не изменяем чистоту модуля калькулятора.
  3. Документируем входные/выходные параметры в начале файла комментариями.

Соблюдение этих правил гарантирует единый стиль и облегчает поддержку проекта на GitHub Pages.
